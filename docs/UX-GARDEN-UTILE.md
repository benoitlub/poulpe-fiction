# UX — Un Garden utile

## Constat

Le Garden montre des structures, états et métaphores, mais il ne répond pas assez vite aux questions essentielles :

- Que fait Gérard maintenant ?
- Qu'a-t-il trouvé ?
- Qu'a-t-il préparé ?
- Qu'est-ce qui a changé ?
- Qu'est-ce qui attend Benoît ?

## Décision

Le Garden devient une vue de situation vivante, pas un inventaire technique.

## Contenu prioritaire

Le premier écran doit afficher au maximum cinq éléments :

1. **Gérard maintenant** — activité réelle ou calme choisi.
2. **Une chose qui pousse** — graine, geste ou aventure prioritaire.
3. **Une récolte prête** — résultat consultable immédiatement.
4. **Une épine** — seul blocage concret demandant Benoît.
5. **Ce que Gérard a appris** — une connaissance ou un geste récent.

Chaque élément doit contenir :

- un titre naturel ;
- une phrase expliquant pourquoi cela compte ;
- une preuve ou provenance ;
- une seule action principale.

## Parcours principal

`Voir Gérard → ouvrir ce qui pousse → voir la récolte → valider uniquement la sortie`

Ne pas obliger l'utilisateur à comprendre Seeds, Sprouts, Packs, Runtime ou workflows pour obtenir le résultat.

## États vides

Un Garden vide ne doit pas être silencieux ou abstrait. Il doit expliquer :

- ce que Gérard observe ;
- pourquoi rien n'est prioritaire ;
- quand le prochain cycle est prévu ;
- quelle source manque éventuellement.

## Règle

La métaphore du Garden doit clarifier l'activité réelle. Elle ne doit jamais cacher l'absence de résultat.
